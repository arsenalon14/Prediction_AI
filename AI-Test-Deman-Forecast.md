# AI Demand Forecasting & Inventory Optimization
## Thai AirAsia (TAA) Buy-on-Board Perishable Model

This document outlines the end-to-end framework for forecasting and optimizing perishable buy-on-board inventory for Thai AirAsia (TAA). It establishes the **DMK→MLE→DMK** pair route as the atomic unit of decision-making, prioritizes the 10 relevant operations research theories, identifies critical data quality issues, and presents an elegant mathematical solution to bypass operational noise.

---

## 1. Operational Context & Decision Unit

For low-cost carrier (LCC) operations, catering replenishment occurs exclusively at the primary hub (**DMK**) to minimize outstation turnaround times and ground service contracts. 

* **Hard Constraint:** All perishable inventory loaded at DMK must be disposed of at the end of the return leg (**MLE→DMK**). 
* **Atomic Unit:** The **pair route (round trip)** is the single correct atomic unit for every forecasting and inventory decision. Leg-by-leg planning is mathematically and operationally incorrect because inventory cannot be replenished at the outstation.

## 3. Core Data Quality Problems

In TAA operations, two fundamental data quality issues corrupt standard forecasting models:

### Problem A: Standard Demand Censoring
When stock sells out, sales drop to zero. The actual customer demand after the stockout is lost (censored). Standard forecasting algorithms trained on raw sales data will systematically under-predict demand, leading to perpetual stockouts.

### Problem B: Crew Misattribution (Logging Errors)
In LCC cabin operations, crew members frequently fail to open/close cart sessions accurately at the outstation (MLE). Consequently, sales transactions from the return sector (**MLE→DMK**) are commonly logged under the outbound sector (**DMK→MLE**), or vice versa. 
* **Impact:** Leg-specific sales data is highly inaccurate, rendering leg-by-leg forecasting useless.

### Problem C: Passenger Manifest Column A Header Anomaly (DictReader KeyError)
In LCC database exports, boarded passenger manifests are frequently generated by varying legacy passenger service systems (PSS). In some reports, Column A is named `FlightDate`, while in others (such as the 2026 manifest `Passenger_Nat_Age_Seat-2026_Month5_24.csv`), it is named `DATE` or `Date`.
* **The Risk:** Strict dictionary readers (like `csv.DictReader` in Python) that query `row['FlightDate']` will fail with an active or silent `KeyError` when headers vary by casing or naming. This skips the entire manifest parsing pipeline, resulting in a **0% demographic matching rate** or program crashes.
* **The Solution:** Avoid hardcoded dictionary key lookups for varying database schemas. Always use a case-insensitive, dynamic index-based column mapper. The loader must scan the first row's headers, match synonyms case-insensitively (e.g. `flightdate`, `date`, `flight_date`), and fall back to **Column A (Index 0)** as the DATE field if no text match is found. This ensures 100% fail-safe manifest parsing.

### Problem D: Outbound-Only Wastage Calculation Flaw (False Spoilage)
In the airline's standard logging, the `wastage` column is derived using the formula `open_quantity - sale_quantity`. However, because 100% of the wastage logs are recorded under the outbound leg (Flight 175) for the entire pair-route session, the `sale_quantity` used in this formula ONLY accounts for items sold on the outbound flight. It completely fails to subtract the items sold on the return leg (Flight 176).
* **The Risk:** This mathematically over-inflates the `wastage` column, falsely logging all food eaten on the return flight (176) as "spoilage/waste". Relying on this column will artificially bloat historical wastage costs and destroy accurate Newsvendor margin structures.
* **The Solution:** Never use the flawed `wastage` column from the raw CSV. The true total cost of loaded perishable food is derived solely from `Total Cost = Sum(open_quantity * unit_cost)`. True physical wastage cost must be manually derived as `Total Cost - Total Pair-Route COGS (Flight 175 + Flight 176)`.

### Problem E: Theory 5 Demographic Scaling — PBM-Ratio Inflation Bug (FIXED 2026-05-29)
The original Theory 5 demographic multiplier contained a mathematical flaw that inflated every Q* recommendation by the PBM prebooking ratio before any demographic effect was applied.
* **Root Cause:** The per-passenger demand rate was computed as `D_0 = mu / N_active` where `N_active = total_pax - pbm_count`. However, the age distribution (`ages` dict) was built from **all** passengers including PBM buyers, so summing `age_count` over all brackets produced a total ≈ `total_pax`, not `N_active`. The denominator and numerator disagreed on whether PBM buyers were in scope.
* **The Impact:** With neutral demographic multipliers (all-Thai Songkran flight at ~30% PBM rate), every item's Q* was inflated by `total_pax / N_active ≈ 1.43` — a 43% over-recommendation on the highest-PBM flights, recreating the spoilage problem the system was designed to solve.
* **The Solution:** Change the base rate to `D_0 = mu / total_pax`. This correctly normalises demand against total passenger count. With neutral multipliers, `mu_adj = mu × weighted_γ × weighted_θ`, scaling only by the demographic mix change, with zero PBM-ratio inflation. This assumes PBM buyers share the same demographic distribution as the full manifest, which is the correct approximation when PBM-specific demographics are not separately tracked.

### Problem F: Net BoB Market Calculation — Cross-File Date Mismatch (FIXED 2026-05-29)
The `/api/upload` endpoint computed `net_bob = total_pax - pbm_count` by subtracting the row counts of two independently processed files without verifying they referred to the same flight date.
* **The Risk:** Uploading a PBM file from a different date (or uploading only one file) produced a meaningless or negative `net_bob` value. Theory 5 then used this as `N_active`, feeding incorrect demographic scaling into the forecast.
* **The Solution:** After parsing both files, the server compares their extracted `flight_date` values. If they diverge, the endpoint returns HTTP 400 with a clear error message: *"Date mismatch: passenger manifest is YYYY-MM-DD but PBM file is YYYY-MM-DD. Upload matching files."* The frontend `alert()` surfaces this directly to the ops team.
* **Additional fix (2026-05-29 rev 2):** Both CSV files were previously written to disk before the date comparison was performed, leaving orphan `pax_YYYY-MM-DD.csv` / `pbm_YYYY-MM-DD.csv` files on mismatch. The upload handler was restructured: both files are parsed into memory first, dates are validated, and only then are they written to disk — so no files persist on a failed upload.

### Problem G: Water-Code Denominator Mismatch — Systematic Q* Underestimation (FIXED 2026-05-29 rev 2)
The `/api/upload` endpoint counted every row in the uploaded PBM CSV as `pbm_count` (`pbm_count = len(df)`), including water-bottle PBM codes (BWFD, BWAK, BWHL, WBHL). The historical training function `_load_net_bob_history()` correctly excluded these same codes when computing `net_bob` per date.
* **Measured Impact:** 37.6% of all 139,332 historical PBM rows are water codes (22,819/60,944 in 2024; 18,836/51,523 in 2025; 9,163/22,842 in 2026 Jan–Apr; 1,571/4,023 in 2026 May). For a flight with 200 pax and 40 PBMs uploaded, correct `net_bob_up = 175` (only 25 food PBMs), but the bug computed `net_bob_up = 160` (all 40 PBMs subtracted). Result: **8–11% systematic Q* underestimation on every forecast.**
* **The Solution:** Filter water codes from the uploaded count: `ssr_col = _col(df, "SSRCode")` → `pbm_count = int((~df[ssr_col].isin(WATER_CODES)).sum())`. This aligns the upload denominator with `_load_net_bob_history()`, removing the train/serve mismatch.

Empirical analysis of TAA's 2024 and 2025 flight manifests and Pre-Booked Meal (PBM) data reveals high-value structural insights that directly inform our forecasting models:

### A. Route Seasonality & Passenger Profiles
* **Summer Peaks (July & August):** Flight volumes reach their annual high of **10,400 to 11,800 passengers/month**. **Chinese tourists** dominate the flights, representing **41% to 54%** of the entire passenger load. **Children (0-12)** peak at **11.8% to 14.1%**, indicating high seasonal demand for child-friendly items.
* **Songkran Peak (April):** **Thai tourists** surge to their highest share (**43.6% to 47.6%**) as domestic leisure travelers head to the Maldives.
* **Winter Low Season (November):** Traffic drops to **6,300 to 7,100 passengers/month**. **Maldives nationals** rise to their highest share (**21% to 25%**), representing steady utility travel (business/medical/shopping).

### B. The PBM Filtering & Booking Discrepancy Rules
* **Water Bottle Exclusion Rule:** Non-perishable items like bottled water (`BWFD`, `BWAK`, `BWHL`, `WBHL`) are filtered out of the perishable forecast. We focus strictly on true perishable hot meals (`VBPM`, `CBCB`, `CTCB`, `PTCB`, `CRCB`).
* **The PBM Booking Discrepancy:** Thai passengers are highly proactive pre-bookers (representing **30% to 69%** of all PBMs). Chinese passengers rarely pre-book (representing only **8% to 20%** of PBMs despite making up half the flight load).
* **Forecasting Implication:** Because the high-volume Chinese tourist segments do not pre-book, **in-flight Buy-on-Board (BoB) cart demand will experience massive surges during July, August, and February.** Accurate BoB stock allocation is critical to prevent severe revenue loss.

## 4. The Solution: Pair-Route Aggregation (S1 + S2 Collapse)

To resolve both problems simultaneously, we implement an elegant modeling breakthrough: **we collapse Sector 1 (S1) and Sector 2 (S2) into a single aggregated pair-route unit.**

```
Traditional (Corrupted):
Sector 1 (DMK-MLE) Sales (Corrupted by Crew Misattribution)  ---> Leg 1 Forecast (Fails)
Sector 2 (MLE-DMK) Sales (Corrupted by Crew Misattribution)  ---> Leg 2 Forecast (Fails)

The Aggregation Solution (Robust):
[ Sector 1 Sales ] + [ Sector 2 Sales ] ===> Total Pair Sales (Y_total) ---> Pair Forecast (Succeeds)
```

### Why This Fix Works:
1. **Renders Misattribution Irrelevant:** By summing the sales of both sectors, we look at the total quantity sold over the round trip ($Y_{\text{total}} = Y_1 + Y_2$). It no longer matters which leg the crew scanned the item on; the total sum is $100\%$ operationally accurate.
2. **Simplifies Censoring Detection:** Leg-specific stockout tracking is notoriously unreliable. By aggregating the pair route, censoring detection collapses into a single binary flag:
   $$\text{Censored (Stockout)} = \begin{cases} 
   1 & \text{if } Y_{\text{total}} \geq Q \\ 
   0 & \text{if } Y_{\text{total}} < Q 
   \end{cases}$$
   If the total quantity sold equals or exceeds the total quantity loaded at DMK, a stockout occurred during the trip, and demand was censored.

## 4.1. The Cold Start Strategy for New Seasonal Menus

When the airline introduces a brand-new seasonal menu item (e.g., swapping out Basil Chicken for a new Seafood dish), standard forecasting fails because there is exactly zero historical data. We solve this using a combined **Product Substitution / Bayesian Updating** strategy:

1. **Product Feature Mapping (Similarity Engine):** The AI categorizes the new item by Protein, Flavor Profile, Cuisine, and Category.
2. **Proxy Baseline Assignment:** The model links the new item to the most mathematically similar historical item (e.g., matching a new spicy chicken dish to the retired "ML NOI Fried Chicken Basil"). This provides a highly accurate Day-1 expected demand baseline ($\mu_{\text{proxy}}$).
3. **High-Margin "Exploration" Load (Newsvendor Theory):** Because we know the new item's Retail Price ($p$) and Unit Cost ($c$) from Day 1, the AI calculates its **Critical Ratio**. High-margin items receive an intentional safety stock buffer. Over-catering heavily during the first 3 days is a strategic necessity—if the new item stocks out, the demand becomes censored, and the model cannot learn its true maximum popularity.
4. **Rapid Bayesian Updating:** As real sales data flows in, the model shifts from the Proxy Baseline to True Data. By Day 8, the proxy baseline is discarded entirely, and the AI optimizes the new item purely on its proven, uncensored Tobit demand curve.

## 4.2. 2-3 Days Prior Operational Forecasting Protocol (Demographic Scaling — Theory T5b)

The Baseline Optimal Load ($Q^*$) from the monthly per-pax Tobit model (T2) reflects the *average* nationality composition historically seen in that calendar month. 2–3 days before a specific flight, the operations team uploads the upcoming flight's passenger manifest. Theory T5b (Demographic Gamma, A1) adjusts Q* based on how the upcoming flight's nationality mix deviates from that month's historical norm.

### How T5b Works (Three Steps)

**Step 1 — Preference Weights (computed once, cached):**
Historical PAX files (seat + nationality + date) are inner-joined to the perishable sales files (seat + SKU + quantity + date) on `FlightDate + seat_number`. This produces a purchase record: *"this seat was occupied by a Chinese passenger, who bought 1 Teriyaki."* Aggregating across all 2024–2026 flights gives:

$$\text{pref}[\text{nationality}][\text{SKU}] = \frac{\text{total units sold to nationality}}{\text{total PAX of that nationality}}$$

**Step 2 — Monthly Reference Composition:**
From the same historical PAX files, for each calendar month the model derives the average nationality fraction:

$$\text{hist\_frac}[\text{month}][\text{nationality}] = \frac{\text{PAX of that nationality in month}}{\text{total PAX in month}}$$

For example, July historical: ~54% China, ~20% Thailand, ~15% Maldives, ~11% others.

**Step 3 — γ per SKU (computed on every upload):**
The upcoming flight's manifest provides its own nationality counts. The gamma multiplier measures deviation from the monthly norm:

$$\gamma[\text{SKU}] = \frac{\sum_{\text{nat}} \text{upcoming\_frac}[\text{nat}] \times \text{pref}[\text{nat}][\text{SKU}]}{\sum_{\text{nat}} \text{hist\_frac}[\text{month}][\text{nat}] \times \text{pref}[\text{nat}][\text{SKU}]}$$

* **γ = 1.0** — upcoming composition identical to historical average for this month → no adjustment.
* **γ < 1.0** — upcoming flight is lighter on buyers of this SKU (e.g. 0% Chinese in July → γ ≈ 0.4 for Teriyaki).
* **γ > 1.0** — upcoming flight is heavier on buyers of this SKU (e.g. peak-Chinese August → load more).

Final load recommendation: $Q^*_{\text{adjusted}} = \lceil \gamma \times Q^*_{\text{baseline}} \rceil$

No floor or ceiling clamp is applied algorithmically. The **manual override input** in the Adjusted Q* table column is the operator's safety valve for extreme cases.

## 5. Mathematical Formulations (Pair-Route Collapsed)

### 1. Collapsed Demand Un-Censoring (Tobit Model)
We model the total round-trip demand $D_{\text{total}} = D_1 + D_2$ as a single random variable following a normal distribution:

$$D_{\text{total}} \sim N(\mu_{\text{total}}, \sigma_{\text{total}}^2)$$

Given the loaded capacity $Q$ at DMK, the observed total sales $Y_{\text{total}}$ is censored at $Q$:

$$Y_{\text{total}} = \min(D_{\text{total}}, Q)$$

Using Maximum Likelihood Estimation (MLE), the log-likelihood function for $N$ historical round-trip flights is:

$$\ln L(\mu_{\text{total}}, \sigma_{\text{total}}) = \sum_{j \in \text{Uncensored}} \ln \left[ \frac{1}{\sigma_{\text{total}}} \phi\left(\frac{Y_{\text{total}, j} - \mu_{\text{total}}}{\sigma_{\text{total}}}\right) \right] + \sum_{j \in \text{Censored}} \ln \left[ 1 - \Phi\left(\frac{Q_j - \mu_{\text{total}}}{\sigma_{\text{total}}}\right) \right]$$

Maximizing this function gives us the **true uncensored parameters** $(\mu_{\text{total}}, \sigma_{\text{total}})$, representing actual customer demand on the round-trip route.

### 2. Collapsed Newsvendor Optimization
With the pair route aggregated, the optimization formulation simplifies from a complex two-period dynamic model to a robust single-decision Newsvendor model.

#### Cost Parameters:
* **Underage Cost ($C_u$):** The lost profit margin of not selling a sandwich on the round trip.
  $$C_u = p - c$$
  *(where $p$ is the average selling price and $c$ is the unit catering cost)*
* **Overage Cost ($C_o$):** The cost of discarding an unsold sandwich at the end of the round trip, plus any disposal penalty ($h$).
  $$C_o = c + h$$

#### The Critical Fractile:
The optimal quantity $Q^*$ to load at DMK is the value that satisfies the target service level:

$$F(Q^*) = \Phi\left(\frac{Q^* - \mu_{\text{total}}}{\sigma_{\text{total}}}\right) = \frac{C_u}{C_u + C_o} = \frac{p - c}{p + h}$$

$$Q^* = \mu_{\text{total}} + z \cdot \sigma_{\text{total}}$$

*(where $z$ is the standard normal score corresponding to the Critical Fractile probability)*

## 6. Python Implementation Blueprint

Below is the code to calculate the true demand parameters and output the optimal load quantity $Q^*$ using the collapsed model:

```python
import numpy as np
from scipy.optimize import minimize
import scipy.stats as stats

def fit_uncensored_demand(y_total, Q_loaded, censored_mask):
    """
    Fits a Tobit model to find the true, uncensored mean and standard deviation 
    of demand for the collapsed pair route.
    """
    def log_likelihood(params):
        mu, sigma = params
        if sigma <= 0:
            return 1e10
        
        # Uncensored flights: use Normal PDF
        pdf_term = stats.norm.logpdf(y_total[~censored_mask], loc=mu, scale=sigma)
        
        # Censored flights (stockouts): use Survival Function (1 - CDF)
        sf_term = stats.norm.logsf(Q_loaded[censored_mask], loc=mu, scale=sigma)
        
        return -(np.sum(pdf_term) + np.sum(sf_term))

    # Initialize guesses using basic sales statistics
    init_mu = np.mean(y_total)
    init_sigma = np.std(y_total) if np.std(y_total) > 0 else 1.0
    
    res = minimize(log_likelihood, x0=[init_mu, init_sigma], method='Nelder-Mead')
    return res.x[0], res.x[1]

def calculate_optimal_uplift(mu_true, sigma_true, price, cost, disposal_penalty=0.0):
    """
    Calculates the optimal load quantity Q* using the Critical Fractile.
    """
    C_u = price - cost
    C_o = cost + disposal_penalty
    
    critical_ratio = C_u / (C_u + C_o)
    
    # Calculate target z-score
    z = stats.norm.ppf(critical_ratio)
    
    # Optimal load recommendation
    Q_opt = mu_true + z * sigma_true
    return max(0, int(np.round(Q_opt)))

# ==========================================
# Example Simulation & Testing
# ==========================================
if __name__ == "__main__":
    # Mock data: 10 historical round-trips
    # Loaded 20 items on all flights
    Q_loaded = np.array([20, 20, 20, 20, 20, 20, 20, 20, 20, 20])
    
    # Sum of S1 + S2 recorded sales
    y_total  = np.array([12, 20, 15, 20, 18, 14, 20, 19, 11, 20]) 
    
    # Determine censoring (stockout occurred)
    censored_mask = y_total >= Q_loaded
    
    # Step 1: Recover true demand distribution
    mu_true, sigma_true = fit_uncensored_demand(y_total, Q_loaded, censored_mask)
    print(f"Observed Mean Sales: {np.mean(y_total):.2f}")
    print(f"True (Uncensored) Expected Demand: {mu_true:.2f} (Std: {sigma_true:.2f})")
    
    # Step 2: Optimize Q* for a sandwich (Price = 120 THB, Cost = 40 THB)
    Q_opt = calculate_optimal_uplift(mu_true, sigma_true, price=120, cost=40)
    print(f"Optimized DMK Uplift recommendation (Q*): {Q_opt} units")

### 10.2. Theory 7 Clarification — MLE Refit vs. Bayesian Updating

The post-flight feedback loop (Theory 7) re-runs the **full Tobit MLE** on the combined historical + actuals dataset. This is frequentist, not Bayesian. A true Bayesian update would apply the prior distribution `(μ_prior, σ_prior)` and shift it proportionally to the weight of new evidence. The MLE refit is equivalent to Bayesian updating only when the prior is flat (uniform). In practice, with 1–2 new observations on top of 140+ historical records, the refit produces nearly identical results to the prior and the difference is negligible. The label was corrected for accuracy. No algorithmic change was made.

## 10.5. A1 Demographic Gamma Implementation (2026-05-29 rev 3)

Theory T5b (Demographic Gamma, A1) was implemented in `app.py` and `forecast_app.html`. The design question resolved before implementation: does the SaleALL CSV contain a seat column that can be joined to `UnitDesignator` in the PAX CSV? Verified: `seat_number` column present in sales, `UnitDesignator` in PAX, both using `YYYY-MM-DD` date format — seat join is fully feasible.

### New Functions Added to `app.py`

| Function | Role |
| :--- | :--- |
| `_build_preference_weights()` | Joins PAX (FlightDate, UnitDesignator, Country_name) to sales (flight_date, seat_number, product_code, quantity). Returns `{country: {sku: avg_units_per_pax}}`. Cached in `_pref_weights_cache`; returns `{}` if join yields < 500 matches. |
| `_get_monthly_nat_fracs(month)` | Returns `{country: fraction}` for historical flights in the given calendar month. Cached in `_monthly_nat_cache` (never invalidated — PAX files are static). |
| `_compute_gamma(nat_counts, month, sku_codes)` | Wraps both above. Normalises `nat_counts` to fractions; computes γ per SKU. If pref or hist data is unavailable for a SKU, γ = 1.0. No clamp applied. |

### Changes to Existing Functions

| Function | Change |
| :--- | :--- |
| `run_pipeline()` | Pre-computes `gammas = _compute_gamma(nat_counts, upcoming_month, sku_list)` once before the item loop. First and proxy passes: `q_baseline = ceil(q_newsvendor)`, `q_adjusted = ceil(q_newsvendor × γ)`, `delta = q_adjusted − q_baseline`. `gamma` field added to each result dict. T5b tag appended when `|γ − 1| > 0.01`. |
| `upload_csv()` | Nationality bucketing changed from `{"TH": n, "international": n}` to top-5 actual country names + `"Others"` (raw counts). Provides granular input for γ computation. |
| `post_flight_actuals()` | Invalidates `_pref_weights_cache = None` so the next forecast re-derives preference weights incorporating the newly uploaded actuals. |

### Changes to `forecast_app.html`

| Change | Detail |
| :--- | :--- |
| Nationality pills | After upload, a row of `nat-pill` spans shows each nationality and its percentage of the manifest (e.g. "China 54.2%"). Replaces the former TH/international grouping. |
| Adjusted Q* editable input | The Adjusted Q* table cell is now a `<input type="number" class="q-override">` instead of plain text. Operator can type any override value. |
| Live Δ update | `input` event listener on each `.q-override` recalculates `val − baseline` and updates the Δ badge class (badge-up / badge-down / badge-neutral) in real time. |
| γ tag | When `|γ − 1| > 0.01`, a small `theory-tag` displaying `γ=0.XX` appears beside the adjusted Q* input as a visual cue that demographic adjustment was applied. |

### Design Decisions

* **No γ clamp**: A flight with 0% Chinese in July legitimately produces γ ≈ 0.4 for Teriyaki. Clamping at [0.5, 2.0] would silently suppress a valid signal. The manual override is the operator's correction mechanism.
* **γ reference is monthly average, not all-time average**: The monthly Tobit μ_rate already encodes the month's historical demographic composition. Using γ = upcoming / monthly_historical avoids double-counting. Using γ = upcoming / all-time would re-apply July's Chinese-tourist premium on top of what μ_rate[July] already reflects.
* **Preference weights cached at session level**: Building the seat join across 835 flight dates and ~200k rows on every forecast request is unnecessary. The cache is invalidated only when new actuals arrive (T7 refit), since actuals add more sales rows that marginally update the preference profile.
