The Problem It Solvesปัญหาที่แก้ไข

Cabin crew scan food on Flight 175 (outbound DMK→MLE) and Flight 176 (return MLE→DMK), but regularly log sales under the wrong leg. Leg-by-leg data is corrupted and completely unusable for forecasting.

ลูกเรือสแกนอาหารบนเที่ยวบิน 175 (ขาออก DMK→MLE) และ 176 (ขากลับ MLE→DMK) แต่มักบันทึกยอดขายผิดขา ข้อมูลแยกตามขาเสียหายและใช้คาดการณ์ไม่ได้

What the Code Doesสิ่งที่โค้ดทำ

For every historical flight date and every product, it adds Flight 175 sales + Flight 176 sales into a single number: the total sold on the round trip. Crew scanning errors become mathematically irrelevant.

สำหรับทุกวันที่เที่ยวบินในประวัติและทุกสินค้า นำยอดขายเที่ยวบิน 175 + เที่ยวบิน 176 รวมกันเป็นตัวเลขเดียว คือยอดรวมต่อรอบเดินทาง ข้อผิดพลาดของลูกเรือจึงไม่มีผลทางคณิตศาสตร์

Key Insightข้อสรุปสำคัญ

All food is loaded at Bangkok (DMK) and disposed of at Bangkok on the return. The round trip — not each leg — is the correct planning unit for catering decisions.

อาหารทั้งหมดบรรจุที่กรุงเทพ (DMK) และทิ้งที่กรุงเทพเมื่อกลับมา รอบเดินทาง — ไม่ใช่แต่ละขา — คือหน่วยวางแผนที่ถูกต้องสำหรับการจัดเตรียมอาหาร

The Problem It Solvesปัญหาที่แก้ไข

When food runs out mid-flight, the sales record stops at zero — but passengers were still hungry. Raw sales data permanently underestimates true demand. Training on raw sales locks the system into perpetual stockouts.

เมื่ออาหารหมดกลางเที่ยวบิน บันทึกยอดขายหยุดที่ศูนย์ — แต่ผู้โดยสารยังต้องการอาหารอยู่ ข้อมูลยอดขายดิบประเมินอุปสงค์จริงต่ำเกินไปถาวร ฝึกโมเดลจากยอดขายดิบจะทำให้สินค้าหมดซ้ำแล้วซ้ำเล่า

What the Code Doesสิ่งที่โค้ดทำ

Using Tobit Maximum Likelihood Estimation (MLE), the code fits a demand curve using both sold-out flights (censored observations) and normal flights together. It recovers the true average demand rate per passenger (μ_rate) and variability (σ_rate) — separately for each calendar month, using 835 historical pair-route dates.

ใช้ Tobit Maximum Likelihood Estimation (MLE) โค้ดฟิตเส้นโค้งอุปสงค์จากทั้งเที่ยวบินที่ขายหมด (ข้อมูลถูกตัด) และเที่ยวบินปกติร่วมกัน กู้อัตราอุปสงค์จริงต่อผู้โดยสาร (μ_rate) และความผันแปร (σ_rate) แยกตามเดือนปฏิทิน โดยใช้ 835 วันคู่เส้นทางในประวัติ

Key Insightข้อสรุปสำคัญ

The AI reverses-engineers what passengers actually wanted, not just what was sold before the cart ran empty. This is what makes the True Demand (μ) column in the Q* table different from raw average sales.

AI ย้อนหาว่าผู้โดยสารต้องการจริงอะไร ไม่ใช่แค่สิ่งที่ขายได้ก่อนรถเข็นหมด นี่คือสิ่งที่ทำให้คอลัมน์ อุปสงค์จริง (μ) แตกต่างจากยอดขายดิบ

The Question This Step Answersคำถามที่ขั้นตอนนี้ตอบ

Before recommending a loading quantity, the system must know: which mistake is more expensive — loading too little (losing a sale) or loading too much (throwing food away)?

ก่อนแนะนำปริมาณการโหลด ระบบต้องรู้ว่าข้อผิดพลาดใดแพงกว่า — โหลดน้อยเกินไป (เสียยอดขาย) หรือโหลดมากเกินไป (ทิ้งอาหาร)?

What the Code Doesสิ่งที่โค้ดทำ

For each product, it reads the retail price and unit cost, then computes two numbers:
Under-catering cost (C_u) = Price − Cost (lost margin per missed sale) Over-catering cost (C_o) = Unit cost (food wasted per unsold meal)

สำหรับแต่ละสินค้า อ่านราคาขายปลีกและต้นทุนต่อหน่วย แล้วคำนวณสองตัวเลข:
ต้นทุนโหลดน้อย (C_u) = ราคา − ต้นทุน (กำไรหายต่อการพลาดขาย) ต้นทุนโหลดเกิน (C_o) = ต้นทุนต่อหน่วย (อาหารเสียต่อมื้อที่ขายไม่ได้)

Key Insight — Basil Chicken Exampleข้อสรุปสำคัญ — ตัวอย่างไก่กะเพรา

Loading too little: loses 102 THB per unsold opportunity (150 − 48 THB).
Loading too much: wastes 48 THB per discarded meal.
Under-catering is 2.1× more expensive than over-catering — so the model skews toward loading more.

โหลดน้อยเกินไป: เสียกำไร 102 บาทต่อโอกาสขาย (150 − 48 บาท)
โหลดมากเกินไป: สูญเสีย 48 บาทต่อมื้อที่ทิ้ง
การโหลดน้อยแพงกว่า 2.1 เท่า ดังนั้นโมเดลจึงเอนเอียงให้โหลดมากขึ้น

What This Step Decidesสิ่งที่ขั้นตอนนี้ตัดสิน

Given the two costs from Step 3, what is the optimal probability of not running out? Loading for 100% certainty means massive waste. Loading for 50% means constant stockouts. The correct balance is mathematically derivable.

จากต้นทุนสองแบบในขั้นที่ 3 ความน่าจะเป็นที่เหมาะสมในการไม่สินค้าขาดคือเท่าไร? โหลดเพื่อ 100% แน่นอนหมายถึงสูญเสียมาก โหลดเพื่อ 50% หมายถึงสินค้าขาดตลอด ส่วนที่สมดุลคำนวณได้ทางคณิตศาสตร์

What the Code Doesสิ่งที่โค้ดทำ

It divides the under-catering cost by the total of both costs:
F* (Service Level) = C_u ÷ (C_u + C_o) = (Price − Cost) ÷ Price
This percentage appears in the Service Level (F*) column as a gold bar. The code then converts it to a z-score — the safety stock buffer above the mean demand.

หารต้นทุนโหลดน้อยด้วยต้นทุนรวม:
F* (ระดับบริการ) = C_u ÷ (C_u + C_o) = (ราคา − ต้นทุน) ÷ ราคา
เปอร์เซ็นต์นี้แสดงในคอลัมน์ ระดับบริการ (F*) เป็นแถบสีทอง แล้วแปลงเป็น z-score — บัฟเฟอร์สต๊อกความปลอดภัยเหนือค่าเฉลี่ยอุปสงค์

Key Insight — Range Across the Menuข้อสรุปสำคัญ — ช่วงตลอดเมนู

Service levels range from 51% (Burnt Cheesecake, lower margin) to 68.5% (Green Curry, highest margin). Higher margin = system recommends more safety stock to capture more of those sales.

ระดับบริการอยู่ในช่วง 51% (เค้กชีสไหม้, กำไรต่ำ) ถึง 68.5% (แกงเขียวหวาน, กำไรสูงสุด) กำไรสูงกว่า = ระบบแนะนำสต๊อกความปลอดภัยมากขึ้นเพื่อดักจับยอดขาย

The Requirementข้อกำหนด

Steps 2–4 produce a demand rate per passenger. The Q* table must show a specific number of items to physically load. This requires knowing how many passengers on this specific flight will buy from the cart.

ขั้นที่ 2–4 สร้างอัตราอุปสงค์ต่อผู้โดยสาร ตาราง Q* ต้องแสดงจำนวนสินค้าที่โหลดจริง ซึ่งต้องรู้ว่าผู้โดยสารกี่คนในเที่ยวบินนี้จะซื้อจากรถเข็น

What the Code Doesสิ่งที่โค้ดทำ

It reads the uploaded passenger manifest and computes Net BoB (passengers who will buy on-board):
Net BoB = Total pax boarded − Pre-booked meal pax Baseline Q* = (μ_rate × Net BoB) + z × (σ_rate × Net BoB)

อ่านรายชื่อผู้โดยสารที่อัปโหลด และคำนวณ Net BoB (ผู้โดยสารที่จะซื้อบนเครื่อง):
Net BoB = ผู้โดยสารที่ขึ้นเครื่อง − ผู้โดยสารที่จองอาหารล่วงหน้า Baseline Q* = (μ_rate × Net BoB) + z × (σ_rate × Net BoB)

Worked Example — Boba Tea, May Flightตัวอย่างจริง — ชาไทยบีบ้า เดือนพฤษภาคม

173 net BoB passengers. μ_rate for May = 0.107 units/pax.
Expected demand = 0.107 × 173 = 18.5 units.
With z safety buffer → Baseline Q* = 21. That's the number in the table.

ผู้โดยสาร Net BoB 173 คน μ_rate เดือนพฤษภาคม = 0.107 หน่วย/คน
อุปสงค์ที่คาดหวัง = 0.107 × 173 = 18.5 หน่วย
รวม z safety buffer → Baseline Q* = 21 นั่นคือตัวเลขในตาราง

The Refinementการปรับปรุงเพิ่มเติม

Step 5's Baseline Q* uses the historical average nationality mix for that calendar month. But each specific flight may have a very different composition — and different nationalities buy very different items.

Baseline Q* ของขั้นที่ 5 ใช้สัดส่วนสัญชาติค่าเฉลี่ยประวัติของเดือนนั้น แต่แต่ละเที่ยวบินอาจมีองค์ประกอบต่างกันมาก และสัญชาติต่างกันซื้อสินค้าต่างกันมาก

What the Code Doesสิ่งที่โค้ดทำ

It reads the actual upcoming manifest nationality counts and computes a multiplier per item:
γ = This flight's predicted demand rate ÷ Historical monthly average rate Adjusted Q* = ⌈ Baseline Q* × γ ⌉ γ is learned from 248,752 historical passenger seat records joined to 177,173 sales transactions.

อ่านจำนวนสัญชาติจากรายชื่อผู้โดยสารจริง แล้วคำนวณตัวคูณต่อสินค้า:
γ = อัตราอุปสงค์คาดการณ์เที่ยวบินนี้ ÷ อัตราเฉลี่ยรายเดือนประวัติ Adjusted Q* = ⌈ Baseline Q* × γ ⌉ γ เรียนรู้จากข้อมูลที่นั่ง 248,752 รายการ จับคู่กับยอดขาย 177,173 รายการ

Key Exampleตัวอย่างสำคัญ

July average: 54% Chinese passengers (heavy Teriyaki buyers). If a July flight has 0% Chinese, γ ≈ 0.40 for Teriyaki → load 40% less than baseline. The γ=0.82 tag visible in the Q* table is this multiplier.

ค่าเฉลี่ยกรกฎาคม: ผู้โดยสารจีน 54% (ซื้อเทอริยากิมาก) หากเที่ยวบินกรกฎาคมมีจีน 0% γ ≈ 0.40 สำหรับเทอริยากิ → โหลดน้อยกว่า baseline 40% แท็ก γ=0.82 ในตาราง Q* คือตัวคูณนี้

The Challengeความท้าทาย

When the airline introduces a new seasonal item, there is exactly zero historical data for it. Steps 2–5 cannot run with no observations. Without a recommendation, the operations team guesses — creating waste or stockouts from Day 1.

เมื่อสายการบินเปิดตัวสินค้าฤดูกาลใหม่ จะมีข้อมูลประวัติเป็นศูนย์ ขั้นที่ 2–5 ทำงานไม่ได้หากไม่มีข้อมูล ทีมปฏิบัติการจะเดา — สร้างของเหลือหรือสินค้าหมดตั้งแต่วันแรก

What the Code Doesสิ่งที่โค้ดทำ

It matches the new item to the most similar established item (the "proxy") by protein type, flavor profile, and price bracket. It borrows that proxy's μ_rate and σ_rate, then runs Steps 3–5 with the new item's own price and cost. The proxy is dropped entirely by Day 8 of real sales.

จับคู่สินค้าใหม่กับสินค้าที่มีข้อมูล (proxy) ที่ใกล้เคียงที่สุดตามโปรตีน รสชาติ และราคา ยืม μ_rate และ σ_rate ของ proxy แล้วทำขั้นที่ 3–5 ด้วยราคาและต้นทุนของสินค้าใหม่เอง Proxy ถูกยกเลิกเมื่อมียอดขายจริงครบ 8 วัน

How to Identify It in the Tableวิธีระบุในตาราง

New items show "T6:Substitution(proxy=...)" in the Theories column instead of the normal T2 Tobit tag. This flags to the operator that the recommendation is proxy-based, not yet fully data-driven.

สินค้าใหม่จะแสดง "T6:Substitution(proxy=...)" ในคอลัมน์ Theories แทนแท็ก T2 Tobit ปกติ แจ้งผู้ปฏิบัติการว่าคำแนะนำนี้อ้างอิง proxy ยังไม่ใช่ข้อมูลจริงครบถ้วน

Why This Step Existsทำไมขั้นตอนนี้จึงมีอยู่

A model trained only on historical data slowly becomes stale. Passenger preferences shift, new items mature, seasonal patterns evolve year to year. Without a feedback loop, Q* recommendations would drift away from current reality.

โมเดลที่ฝึกจากข้อมูลประวัติเท่านั้นจะค่อย ๆ ล้าสมัย ความชอบของผู้โดยสารเปลี่ยน สินค้าใหม่เติบโต รูปแบบฤดูกาลพัฒนาไปในแต่ละปี หากไม่มีวงรอบข้อมูลย้อนกลับ ค่า Q* จะค่อย ๆ เบี่ยงออกจากความเป็นจริง

What the Code Doesสิ่งที่โค้ดทำ

After a flight lands, the operations team uploads three files: actual SaleALL CSV, Wastage CSV, and final boarded passenger manifest. The system appends these permanently and re-runs the full Tobit MLE (Step 2) on all combined data. The next forecast uses the updated μ_rate and σ_rate immediately.

หลังเที่ยวบินลงจอด ทีมปฏิบัติการอัปโหลดสามไฟล์: SaleALL CSV จริง, Wastage CSV, และรายชื่อผู้โดยสารขึ้นเครื่องจริง ระบบบันทึกถาวรและรัน Tobit MLE ใหม่ (ขั้นที่ 2) บนข้อมูลรวมทั้งหมด การคาดการณ์ถัดไปใช้ μ_rate และ σ_rate ที่อัปเดตแล้วทันที

Key Insightข้อสรุปสำคัญ

With 140+ historical records per product, each new flight shifts the demand estimate by a small fraction. The system is self-correcting — it gets marginally smarter after every flight, automatically, with no manual model updates required.

ด้วยข้อมูลประวัติกว่า 140 รายการต่อสินค้า แต่ละเที่ยวบินใหม่ปรับค่าอุปสงค์เล็กน้อย ระบบแก้ไขตัวเอง — ฉลาดขึ้นเล็กน้อยหลังทุกเที่ยวบิน โดยอัตโนมัติ ไม่ต้องอัปเดตโมเดลด้วยมือ

How to Use the System วิธีใช้ระบบ

Pre-Flight & Post-Flight Operational Workflow ขั้นตอนปฏิบัติการก่อนและหลังเที่ยวบิน

✈ Pre-Flight (2–3 Days Before) ✈ ก่อนเที่ยวบิน (2–3 วันล่วงหน้า)
1

Export upcoming flight's passenger manifest CSV from PSS system

ส่งออก CSV รายชื่อผู้โดยสารของเที่ยวบินจากระบบ PSS

2

Export pre-booked meal (PBM) booking data CSV for the same flight date

ส่งออก CSV ข้อมูลการจองอาหารล่วงหน้า (PBM) สำหรับวันเที่ยวบินเดียวกัน

3

Upload both in the Forecast tab — system validates the dates match, returns pax count and demographic breakdown

อัปโหลดทั้งสองในแท็บ Forecast — ระบบตรวจสอบวันที่ตรงกัน คืนจำนวนผู้โดยสารและโครงสร้างสัญชาติ

4

Click "Run Hyper-Personalized Forecast" — all 7 steps execute; Q* appears per active menu item in under 1 second

กด "Run Hyper-Personalized Forecast" — ทำ 7 ขั้นตอน ค่า Q* ปรากฏต่อสินค้าในเมนูในน้อยกว่า 1 วินาที

5

Load catering at DMK per the Adjusted Q* column; override any value if operational context requires it

โหลดอาหารที่ DMK ตามคอลัมน์ Adjusted Q* แก้ไขค่าใดก็ได้หากบริบทปฏิบัติการกำหนด

🛬 Post-Flight (After Return to DMK) 🛬 หลังเที่ยวบิน (หลังกลับถึง DMK)
1

Export actual SaleALL CSV, Wastage CSV, and final boarded passenger manifest from POS/PSS

ส่งออก SaleALL CSV จริง, Wastage CSV, และรายชื่อผู้โดยสารขึ้นเครื่องจริงจาก POS/PSS

2

Upload all three in the Post-Flight Actuals tab — the boarded passenger file is required (system blocks without it)

อัปโหลดทั้งสามในแท็บ Post-Flight Actuals — ไฟล์ผู้โดยสารที่ขึ้นเครื่องจำเป็นต้องมี (ระบบบล็อกหากไม่มี)

3

System saves all files permanently, then re-runs Tobit MLE (Theory 7) on the full combined dataset automatically

ระบบบันทึกไฟล์ถาวร แล้วรัน Tobit MLE ใหม่ (ทฤษฎี 7) บนข้อมูลรวมทั้งหมดอัตโนมัติ

4

Review updated μ, σ, Q* in the MLE Refit results table — the model has incorporated this flight and is ready for the next forecast

ตรวจสอบ μ, σ, Q* ที่อัปเดตในตารางผล MLE Refit — โมเดลผนวกเที่ยวบินนี้แล้ว พร้อมสำหรับการคาดการณ์ถัดไป
